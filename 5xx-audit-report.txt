========================================================================
ROOT CAUSE (diagnosed with live evidence)
========================================================================
These six route families are the ONLY routes in the app that use a
PARTIAL dynamic segment — literal text mixed with a [param] inside a
single path segment, e.g. folder "visa-free-countries-for-[nationality]-passport".

Every route that uses a CLEAN full-segment param works perfectly:
  app/blog/[slug]                      -> 200 / clean 404 on bad slug
  app/visa/[passport]/[destination]    -> 200
  app/destinations/[country]           -> 200

Evidence the page handler NEVER runs for the broken families:
  - Response header x-matched-path: /500  (matched the error page, not the route)
  - x-vercel-id shows EDGE region only (bom1), no function region (iad1).
    A working dynamic page shows bom1::iad1 (function executed).
  - Zero serverless runtime logs for these requests.
  - A brand-new INVALID param still returns 500 (a running handler would
    return a clean 404 via notFound()).  blog[slug] correctly 404s.
  - Deploy lambdaRuntimeStats = only 3 node lambdas total.

Ruled OUT (verified, not assumed):
  - Supabase data: healthy. 372 rows for Pakistan, zero null country_name,
    seo_page_content table present, no query errors.
  - Page code logic: sound; identical safe pattern to the working /visa route.
  - force-dynamic: already present on these files (committed in e9c6cbb).
  - generateStaticParams: NOT the cause — blog/[slug] also uses it and works.
  - Build: succeeds with NO prerender errors (checked full build log, 38k lines).

Conclusion: Next.js 16.2.6 + Turbopack produces broken route output for
partial-dynamic-segment folders (the function/prerender is never wired up,
so Vercel serves the static /500 for any matching path). This matches the
documented class of Turbopack-only build/output bugs in Next 16
(see vercel/next.js#94456 "fails on turbopack, works on webpack").

FINAL FIX (commit 5e12007) — preserves every public URL
========================================================================
Root-caused as: Next.js App Router does not reliably serve route folders
whose name mixes literal text with a [param] in ONE path segment.

Fix: keep the exact public URLs, but serve them from CLEAN nested [param]
routes (the shape that already works for /visa/[passport]/[destination],
/blog/[slug], /destinations/[country]):
  - Added 6 clean routes under app/seo/* (visa-free, cheapest, guide, req,
    req-nat, route) — verbatim copies of the page code, just relocated.
  - Added next.config.mjs `beforeFiles` rewrites mapping each pretty URL
    to its /seo/* route. Public URLs are UNCHANGED (Google still sees
    /visa-free-countries-for-pakistani-passport, etc.).
  - All 7 rewrite patterns validated against Next's bundled path-to-regexp
    (correct capture for multi-hyphen values; no cross-matching).
  - Also fixed the internally-linked /[passport]-to-[destination]-visa-
    requirements route (Template B), same bug.
  - The old partial folders remain on disk (the mount blocks deletion) but
    are dead — beforeFiles intercepts before the filesystem. Safe to delete
    later.
NOTE: build remains on `next build --webpack` (green; unchanged from e034ba2).
